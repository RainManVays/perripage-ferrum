## 0. Главный архитектурный вывод: форк библиотеки, скорее всего, не нужен

`peripage.Printer` уже публично экспортирует низкоуровневые примитивы:

```python
def tellPrinter(self, byteseq: bytes) -> None: ...      # отправить сырые байты без ответа
def askPrinter(self, byteseq: bytes, recv_size=1024) -> bytes: ...  # отправить и получить ответ
```

Это значит, что **почти все** новые команды, найденные при разборе официального приложения,
можно реализовать прямо в PeriPrint — новым методом `PeripageClient`, который берёт
`self._printer` (уже подключённый `peripage.Printer`) и зовёт `tellPrinter(bytes.fromhex(...))`
напрямую, **не трогая и не форкая сам пакет `peripage`**. Форк/PR в апстрим становится
самостоятельной, отдельной по срочности целью ("поделиться с сообществом"), а не блокером для
использования этих возможностей в PeriPrint.

Единственное, что **не** обходится простым вызовом `tellPrinter()` — это переписывание того,
как `Printer.printRowBytesList()` кодирует высоту чанка (см. Фазу 4): там нужно либо
монки-патчить/подклассить `Printer`, либо просто не вызывать этот метод библиотеки вообще и
слать чанки самостоятельно собранными кадрами через тот же `tellPrinter()`. Тоже без форка.

Из всех пунктов ниже реальным кандидатом на PR в апстрим (после локальной проверки на железе)
выглядят: расширенный диапазон `setConcentration()`, честная 16-битная высота в
`printRowBytesList()` (закрывает как раз собственный комментарий автора библиотеки про лимит
`0xff`), и документирование новых опкодов через дополнительные методы — это всё
самодостаточные, полезные всем пользователям библиотеки изменения, без привязки к
специфике PeriPrint.

## 2. Фаза 1 — слушатель асинхронных статусов принтера (наивысший приоритет)

**Почему это приоритет №1**: это единственный пункт, где отсутствие функциональности — не
"печатаем чуть хуже", а "приложение не узнаёт, что принтер попросил остановиться" (см.
протокольный документ §4 — входящие `fd01`/`fd02` это ровно "прервать"/"продолжить"). Сейчас
`PrintJobManager._process_job()` (`services/job_manager.py:162`) слепо шлёт чанк за чанком и
узнаёт о проблеме только через исключение при записи в сокет — не через то, что сам принтер
уже сообщил о проблеме.

### Что добавить

Новый класс, например `infra/printer_status_listener.py`:

```python
import os
import select
import threading

# Теги/подтеги разобраны в docs/bluetooth-protocol-trace-analysis.md §7.3
_STATUS_MEANINGS = {
    (0xFF, 1): "out_of_paper",
    (0xFF, 2): "cover_open",
    (0xFF, 3): "overheat",
    (0xFF, 4): "low_battery",
    (0xFF, 5): "cover_closed",
    (0xFF, 6): "low_mileage",       # см. research.md про связь с A3SDK/расходниками
    (0xFD, 1): "abort_print",       # принтер сам просит прервать
    (0xFD, 2): "resume_print",      # принтер разрешает продолжить
    # 0xFE — не просто статус, второй байт это код типа бумаги (см. ниже)
}

class PrinterStatusListener:
    """Фоновый поток, читающий спонтанные 2-байтовые статусные пакеты с
    того же файлового дескриптора, что использует peripage.Printer, не
    мешая обычной отправке данных (полный дуплекс, независимые read/write)."""

    def __init__(self, sock, on_event):
        self._fd = sock.fileno()
        self._on_event = on_event
        self._stop = threading.Event()
        self._thread = threading.Thread(target=self._run, daemon=True)

    def start(self) -> None:
        self._thread.start()

    def stop(self) -> None:
        self._stop.set()

    def _run(self) -> None:
        while not self._stop.is_set():
            ready, _, _ = select.select([self._fd], [], [], 0.2)
            if not ready:
                continue
            try:
                data = os.read(self._fd, 2)
            except OSError:
                return
            if len(data) != 2:
                continue
            tag, sub = data[0], data[1]
            if tag == 0xFE:
                self._on_event(("paper_type_mismatch", sub))
            else:
                meaning = _STATUS_MEANINGS.get((tag, sub))
                if meaning:
                    self._on_event((meaning, sub))
```

Важные оговорки, честно перенесённые из протокольного документа (не всё проверено на живом
трейсе — см. `bluetooth-protocol-trace-analysis.md` §7.3 за статусом уверенности каждого
пункта):

- Формат "тег + подтег по 1 байту" подтверждён декомпиляцией; **не все** пары `(tag, sub)`
  видел живой трейс — только `fd01`/`fd02`. Остальные (`ff01`..`ff06`) подтверждены только по
  коду приложения, не проверялись на реальном принтере отдельно — при первой интеграции стоит
  логировать сырые байты, а не молча полагаться на таблицу.
- Чтение с того же `fd`, что патчится в `_patch_socket_send_recv()`
  (`infra/peripage_client.py:25`) — слушатель нужно запускать **после** этого патча (после
  `printer.connect()`), на том же raw `fd`, а не через `printer.sock.recv()` (тот же паттерн,
  что уже используется в проекте для обхода PyBluez-бага).
- Реакция на `abort_print`: `PrintJobManager` должен прервать текущий цикл отправки чанков
  (в `_process_job()`), выставить `JobStatus.PAUSED_ERROR` и **не** пытаться слать оставшиеся
  чанки в этот же цикл — принтер уже сказал, что не готов их принимать.
- В `services/events.py` нужен новый `EventType.PRINTER_STATUS` (по аналогии с уже
  существующими `PRINT_ERROR`/`CONNECTION_STATUS`), чтобы прокинуть событие в UI
  (`ui/main_window.py`/`ui/queue_panel.py`) — сейчас в приложении вообще нет канала для
  "принтер сообщает о состоянии сам по себе", не в ответ на запрос.

## 3. Фаза 2 — снять искусственный клэмп концентрации (низкий риск)

`peripage.Printer.setConcentration()` (`__init__.py:474`) ограничивает вход до `{0,1,2}`.
Декомпилированный код приложения (`P(int)`, см. протокольный документ §7.2) не содержит
клэмпа вообще; отдельно, в заголовке legacy-протокола картинки (`C(Bitmap,int)`) официальное
приложение клэмпит значение на `0..3` — то есть диапазон минимум на единицу шире
задокументированного библиотекой.

**План:** не трогая `peripage`, добавить в `PeripageClient` метод, обходящий клэмп:

```python
def set_concentration_raw(self, value: int) -> None:
    """Обходит клэмп 0-2 в peripage.setConcentration(): де-факто прошивка
    принимает как минимум 0-3 (подтверждено кодом приложения), возможно
    больше (в трейсе увидено значение 4). Не проверено на предмет визуальной
    разницы в плотности печати — только на предмет того, что команда
    вообще проходит без ошибки."""
    if self._printer is None:
        raise RuntimeError("Not connected")
    value = max(0, min(255, value))
    self._printer.tellPrinter(bytes.fromhex("10ff1000") + bytes([value]))
```

**Проверка на реальном A40**: напечатать одно и то же тестовое изображение с `concentration=2`
(текущий дефолт PeriPrint, `peripage_client.py:57`) и `concentration=3`/`4`, сравнить плотность
печати и нагрев визуально — то же самое, чем проект уже занимался на этапе 0 (см.
`docs/hardware-notes.md`, находка про `concentration=2, delay=0.05`). Если `3`/`4` дают
уверенно более тёмный/лучший результат без побочных эффектов — поднять `_concentration`
дефолт и/или дать выбор в UI (`SettingsDialog`), при этом **не убирая** имеющийся клэмп у
самого `peripage`, раз он всё равно обходится через `tellPrinter()`.

## 4. Фаза 3 — тип бумаги (новая возможность, отсутствует полностью)

`10ff1003`+байт, метод `B(int)` в приложении (см. протокольный документ §7.2), с логом
`"choosePaperType"`. Значения байта по контексту `onPaperError`-обработчика (§7.3): `1`=
складная с чёрной меткой, `2`=рулонная непрерывная, `3`=самоклеящаяся с зазором, `4`=
перфорированная.

```python
class PaperType(IntEnum):
    FOLDED_BLACK_MARK = 1
    CONTINUOUS_ROLL = 2
    ADHESIVE_GAP = 3
    PERFORATED = 4

def choose_paper_type(self, paper_type: "PaperType") -> None:
    if self._printer is None:
        raise RuntimeError("Not connected")
    self._printer.tellPrinter(bytes.fromhex("10ff1003") + bytes([int(paper_type)]))
```

Актуальность для A40 конкретно (обычная рулонная термобумага) — под вопросом: скорее всего
это нужно моделям, которые печатают на этикетках/наклейках с разными типами разметки, не на
типовом чековом рулоне. Стоит реализовать (дёшево), но не приоритизировать высоко для
текущего железа проекта, если только не появятся жалобы на определение размера/типа бумаги.

## 5. Фаза 4 — убрать искусственный лимит чанка в 255 строк

Собственный комментарий библиотеки (`__init__.py:715-717`, докстринг
`printRowBytesList()`): *"This printer supports pages up to `0xffff` rows, but current
implementation relies on chunked data with height limit of `0xff`"* — автор библиотеки уже
знал, что это ограничение искусственное (одним байтом кодируется высота), не следствие
реальной прошивки. Декомпиляция официального приложения это подтверждает напрямую: и в
legacy-, и в новом протоколе поле высоты — честные 16 бит (см. протокольный документ §7.4).

**Не требует форка `peripage`** — PeriPrint может собирать и слать чанк-заголовок сам, минуя
`printRowBytesList()`, через `tellPrinter()`:

```python
def print_rows_16bit_height(self, rows: list[bytes], row_bytes: int) -> None:
    """Замена peripage.Printer.printRowBytesList() с честной 16-битной
    высотой вместо искусственного лимита в 255 строк на чанк. Байтовый
    порядок — little-endian, как в legacy-протоколе приложения (см.
    bluetooth-protocol-trace-analysis.md §7.4) — НЕ путать с big-endian
    в новом 0x1f-протоколе (Фаза 5)."""
    height = len(rows)
    self._printer.reset()
    header = (
        bytes.fromhex("1d763000")
        + bytes([row_bytes % 256, row_bytes // 256])
        + bytes([height % 256, height // 256])
    )
    self._printer.tellPrinter(header)
    for row in rows:
        self._printer.tellPrinter(row.ljust(row_bytes, b"\0")[:row_bytes])
```

**Важно перепроверить перед внедрением**: библиотечный `printRow()`/`printRowBytesList()`
пишет заголовок как `1d763000 + row_bytes(1 байт) + 00 + height(1 байт) + 00` — то есть у
библиотеки, судя по всему, порядок полей *row_bytes затем padding затем height затем padding*,
и она предполагает `row_bytes` тоже помещается в 1 байт (верно для всех 4 моделей —
максимум 231 у A40+). У приложения (`C()`) порядок иной: `row_bytes` сразу два байта
little-endian, затем `height` два байта little-endian, без отдельных padding-байт между ними
— расположение полей нужно свести аккуратно и **проверить печатью реального многострочного
изображения >255 строк одним чанком**, а не полагаться только на анализ кода.

**Практический эффект**: удаление части логики чанкования в `DocumentPipeline`/
`PrintJobManager`, которая сейчас режет длинные документы на части только из-за этого лимита
(см. `services/pipeline.py`) — меньше `reset()`/пауз между чанками, потенциально быстрее и
чище печать длинных чеков.

## 6. Фаза 5 — новый протокол картинки (`0x1f`, zlib) как альтернативный быстрый путь

Полная спецификация — в протокольном документе §3. Кратко: заголовок из 10 байт (опкод,
резерв, `row_bytes` big-endian×2, `height` big-endian×2, длина сжатых данных big-endian×4),
затем `zlib.compress(raw)[2:]` (без 2-байтового zlib-заголовка).

```python
import zlib

def print_image_fast(self, packed_1bpp: bytes, width_px: int, height_px: int) -> None:
    """Экспериментальная реализация протокола 0x1f, которым официальное
    приложение печатает на A40-классе принтеров (см.
    bluetooth-protocol-trace-analysis.md §3). НЕ проверено на реальном
    принтере с данными, произведёнными не официальным Code.code() —
    см. предостережение ниже перед использованием в проде."""
    row_bytes = (width_px + 7) // 8
    compressed = zlib.compress(packed_1bpp, 6)[2:]  # срезать 2-байтовый zlib-заголовок
    header = (
        b"\x1f\x00"
        + row_bytes.to_bytes(2, "big")
        + height_px.to_bytes(2, "big")
        + len(compressed).to_bytes(4, "big")
    )
    self._printer.tellPrinter(header + compressed)
```

`packed_1bpp` — та же 1-бит-на-пиксель MSB-first упаковка, что уже используется в
`DocumentPipeline` для legacy-протокола (чёрное = 1 при среднем RGB < 190, см. протокольный
документ §7.4) — рендеринг/дизеринг менять не нужно, меняется только то, как готовые байты
упаковываются в сетевой кадр.

**Статус — ПРОВЕРЕНО НА ЖЕЛЕЗЕ 2026-07-15, ПОДТВЕРЖДЕНО СЛОМАНО. Не использовать.**

Реализация была добавлена в `PeripageClient.print_image_fast()` и протестирована на реальном
A40 (`PPG_A40_34C4`) с тестовым изображением (300 строк, ширина принтера, полосатый паттерн).
Результат: **принтер завис и выключился** при получении данных. После ручного перезапуска
принтер полностью восстановился — заряд батареи читается нормально (66%), тестовая печать
через уже проверенный `print_image_no_height_limit()` (Фаза 4, legacy-протокол) прошла без
замечаний сразу после. То есть краш локализован именно в обработке прошивкой данных `0x1f`,
произведённых нашим `zlib.compress()` — не аппаратная неисправность и не разовый сбой связи.

Причина не установлена. Ведущая гипотеза (не проверена): `zlib.compress()` по умолчанию
использует 32КБ-окно (`wbits=15`), а embedded-декомпрессор прошивки может ожидать существенно
меньшее окно и не умеет на это корректно реагировать (падает вместо отказа с ошибкой). Заголовок
(опкод/`row_bytes`/`height`/`payload_len`) подтверждён побайтово корректным по декомпилированному
коду приложения (см. протокольный документ §3.1) — проблема именно в потоке сжатых данных, не в
том, как мы его оборачиваем.

**Прежде чем возвращаться к этой фазе**: либо (а) попробовать `zlib.compressobj(wbits=...)` с
явно меньшим окном и повторить тест на железе осторожно (маленькое изображение, готовность к
ещё одному зависанию), либо (б) снять ещё один живой HCI-трейс официального приложения на ЭТОМ
ЖЕ тестовом изображении и побайтово сравнить его сжатые данные с нашими — не полагаться только
на "тот же алгоритм должен дать тот же результат". Реализация оставлена в кодовой базе как
задокументированный тупик (не вызывается ни из одного рабочего пути) — см. докстринг метода в
`infra/peripage_client.py`.

Экономия трафика (13-14× меньше байт на провод для разреженных изображений), которая была
основным потенциальным плюсом этой фазы, остаётся неактуальной, пока протокол не будет
безопасно воспроизведён.

## 7. Фаза 6 — не приоритизировано, но задокументировано

- `10ff81`+байт (скорость/накал печати, клэмп 80 либо 120 по модели, методы `j()`/`v()`) —
  дёшево добавить по аналогии с Фазой 3, эффект на печать неизвестен без теста на железе.
- `10ff20ee`+байт и его 32-байтный hex-ответ (см. протокольный документ §2 шаг 1) — семантика
  параметра не найдена даже в декомпиляции (нет вызывающего кода в разобранных классах);
  требует либо более глубокого поиска места вызова в `g5.f`/презентерах, либо экспериментов
  на живом принтере (посылать разные значения байта, смотреть на реакцию/ответ).
  Похоже не обязательно для печати (наша библиотека работает без него).
- Пакет `p9` (параллельная реализация транспорта для другого семейства устройств приложения) —
  не разобран; актуально, только если PeriPrint когда-нибудь станет поддерживать
  устройства вне линейки термопринтеров-чеков A6/A40.
- `H()` — альтернативная 5-бит-на-пиксель упаковка для полутонового/фото-режима — не
  разобрана подробно, не нужна для чёрно-белой печати документов.

## 8. Апстрим: что имеет смысл предложить автору `peripage`

Если решено оформлять PR в https://github.com/ — репозиторий автора `bitrate16`/`pegasko`
(проверить актуальный URL перед PR, в `pyproject.toml`/`PyPI`-странице пакета) — кандидаты,
самодостаточные и полезные любому пользователю библиотеки, не только PeriPrint:

1. **Честная 16-битная высота в `printRowBytesList()`/`printRow()`** (Фаза 4) — закрывает
   собственный TODO-комментарий автора про лимит `0xff`, обратно совместимо (старый вызывающий
   код с высотой ≤255 продолжит работать один-в-один).
   **Оговорка**: перед PR — самостоятельно проверить у автора/в issue, действительно ли это
   было ограничением 1 байта из-за незнания реального формата, или было осознанным решением
   (например, из соображений надёжности длинных чанков) — не переписывать чужую библиотеку
   вслепую без объяснения находки и её источника (ссылка на этот протокольный документ).
2. **Расширенный диапазон `setConcentration()`** (Фаза 2) — после подтверждения на железе, что
   значения `3`/`4` дают осмысленный, не ломающий печать результат.
3. **`reset()` без 13 лишних нулевых байт** (см. протокольный документ §2 шаг 5/§7.4) — низкий
   приоритет: если лишние байты безвредны (прошивка их игнорирует — что похоже на правду, раз
   текущий `reset()` и так работает на реальном A40 годами использования библиотеки), менять
   рискованно ради чистоты без ясной выгоды. Задокументировать находку в issue, не обязательно
   менять код.

Что **не** стоит предлагать апстриму как есть: новый `0x1f`-протокол и специфичные для
конкретных моделей команды (тип бумаги, скорость) — это расширение поддерживаемых моделей,
логично обсудить отдельно с автором как новую фичу (возможно, за отдельным флагом/классом
принтера), а не как исправление бага в существующем API.
