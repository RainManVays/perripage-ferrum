# Исследование официального Android-приложения PeriPage

**Дата:** 2026-07-15
**Цель:** добраться до реальной логики работы с принтером (протокол, команды, статусы) в
закрытом официальном приложении, которое разработчики Peripage используют сами, и оценить
безопасность самого APK перед тем, как его вообще анализировать/доверять содержимому.

Это вспомогательный документ с методологией и выводами общего характера. Конкретный,
байт-в-байт разбор протокола печати — в `docs/bluetooth-protocol-trace-analysis.md` (дополнен
результатами этого исследования). Пошаговый план внедрения — в
`docs/printer-protocol-implementation-plan.md`.

---

## 1. Какие файлы разбирались — и один ложный след

В процессе всплыли три файла, не все — то, чем казались:

1. **`peripage-xiamen-ilead-tek-co-ltd.apk`** (лежал в корне репозитория) — при распаковке и
   декомпиляции (`jadx`) оказался **не приложением Peripage вообще**, а установщиком
   Android-магазина приложений **Aptoide** (`cm.aptoide.pt`, версия 9.22.5.3). Ни в манифесте,
   ни в ресурсах, ни в единой java-строке нет упоминаний "peripage"/"ileadtek"/"xiamen" — имя
   файла вводит в заблуждение. Похоже на типичный для Aptoide паттерн: сторонняя страница
   "скачать через Aptoide" на самом деле отдаёт клиент самого магазина, а не целевое
   приложение. **Этот файл был отброшен, для протокола бесполезен.**
2. **`com.ileadtek.peripage-312.xapk`** — настоящий APK (в формате `.xapk`, контейнер из
   base+split APK), package `com.ileadtek.peripage`, версия 6.10.6 (build 312). Источник
   загрузки не был указан явно пользователем.
3. **`peripage-base.apk` + `peripage-split_config.arm64_v8a.apk`** — тот же пакет, версия
   6.10.9 (build 319), **выгружены напрямую с реального устройства пользователя** (более
   авторитетный источник, чем файл (2) неясного происхождения). Использованы как основной
   источник для финального разбора; расхождений в протокольной логике между 312 и 319 не
   найдено (см. §3).

Оба реальных архива (`.xapk`, `.apk`) специально добавлены в `.gitignore` (`*.apk`, `*.xapk`) —
не наш код, не нужен для сборки, не должен попадать в репозиторий.

## 2. Методология

- Распаковка контейнеров: `unzip` (сам APK/XAPK — обычный zip).
- Декомпиляция Java/Kotlin-байткода (`classes*.dex`, до 8 файлов на сборку — приложение
  использует multidex): `jadx` 1.5.6, скачан с официальных GitHub-релизов (`skylot/jadx`,
  локальный `~/…/scratchpad`, не система) — на машине не было предустановлено ни `jadx`, ни
  `apktool`.
- Разбор нативных библиотек (`.so`, `arm64-v8a`): `readelf`, `nm`, `strings`, `objdump`
  (последний не понимает AArch64 в имеющейся сборке binutils — вместо него использован
  Python `capstone` (дизассемблер) + `pyelftools` (парсинг ELF/DWARF) в отдельном
  одноразовом venv в `/tmp` (не в системный Python, не в основной `.venv`/`.venv-bt` проекта —
  это чисто аналитический инструмент для этой задачи, а не зависимость PeriPrint).
- Манифест (`AndroidManifest.xml`, бинарный формат): `pyaxmlparser` (тот же одноразовый venv) —
  для точного разбора permissions/exported-компонентов вместо парсинга сырых строк.
- Никаких `sudo`/системных пакетов не устанавливалось — всё через pip в venv или бинарные
  архивы, распакованные в scratch-директорию, по действующему правилу проекта.

## 3. Оценка безопасности (задача пользователя: проверить на "закладки")

Прежде чем разбирать протокол, было явно проверено, не содержит ли реальный APK
(`com.ileadtek.peripage`, версии 312 и 319) вредоносной функциональности. Кратко — **не
найдено ничего похожего на бэкдор**. Подробности:

- **Динамическая загрузка кода**: `DexClassLoader`/`InMemoryDexClassLoader` нигде не
  вызывается из кода `com.ileadtek.*`. Единственное упоминание `com.android.dx` в дереве
  классов — мёртвый код, транзитивно притянутый какой-то тестовой библиотекой, не
  используется прикладным кодом.
- **Exec/root**: ни `Runtime.exec`, ни `ProcessBuilder`, ни проверок `su`/`/system/bin/su` в
  коде приложения нет.
- **JS-мосты**: `addJavascriptInterface` не используется ни разу.
- **Сеть**: единственный собственный API-домен — `https://global.ileadtek.com` (HTTPS, домен
  производителя). Жёстко зашитых IP-адресов (C2-подобных) не найдено — единственная строка,
  похожая на IP (`"1.0.0.0"`), оказалась версией.
- **Разрешения** (полный список: `BLUETOOTH[_ADMIN/_SCAN/_CONNECT/_ADVERTISE]`, wifi-группа,
  `CAMERA`, `RECORD_AUDIO`, `READ/WRITE_EXTERNAL_STORAGE`, `READ_PHONE_STATE`, `BIOMETRIC`,
  `RECEIVE_BOOT_COMPLETED`) — **не запрашивает** `READ_SMS`, `READ_CONTACTS`, `CALL_LOG`.
  Даже если бы в коде было намерение читать SMS/контакты, ОС Android физически не даст этого
  сделать без соответствующего разрешения — а его нет.
- **Экспортируемые компоненты** (`exported=true` в манифесте) — все объяснимы: activity с
  intent-filter'ами "открыть файл в PeriPage" (документы/картинки), плюс callback-activity
  сторонних SDK для OAuth-редиректов (Facebook, LINE, Tencent QQ, Google Sign-In,
  Mob/ShareSDK, оператор-верификация "вход в один клик") — стандартный паттерн, не бэкдор.
- **`libA3SDK.so`** (JNI-класс `com.peripage.a3.A3SDK`, судя по именам функций —
  антиклон/DRM-проверка подлинности принтера) разобран вручную дизассемблером: линкуется
  **только** с `libc`/`libm`/`liblog` — **нет ни сокетов, ни файлового I/O**. Логика чисто
  локальная: `CheckPrinter()` проверяет, что имя Bluetooth-устройства содержит одну из
  подстрок `"386"/"392"/"LTK"/"QR"/"BY"/"IP"/"PeriPage"`; `CheckValidPrinter()` генерирует
  9-байтовый challenge (случайные байты + XOR с константой `0x1019BB5A`), результат gated
  магическим числом `0x04999BF0`. Физически не может ничего никуда передать — работает
  полностью офлайн. Идентичен байт-в-байт (тот же SHA-256) между версиями 312 и 319.
- **`libCode.so`** — при ближайшем рассмотрении (см. §4) оказался обычной сборкой **zlib
  1.2.3** (сжатие изображений для нового протокола печати), не проприетарным/подозрительным
  кодом.

Много балласта (Facebook/Firebase/Flurry/Sentry/Aliyun-видео/EasyAR AR/OCR/ShareSDK/
Mob-верификация) — типичный трекинг-стек китайского консьюмерского приложения, вопрос
приватности (стоит иметь в виду при использовании самого приложения), но не целевая закладка
производителя в смысле "скрытый удалённый доступ к устройству/данным".

## 4. Архитектура протокольного кода приложения

Код сильно обфусцирован (ProGuard/R8, однобуквенные пакеты/классы), но структура
восстанавливается по связям импортов и по строковым Log-тегам, переживающим обфускацию:

- **Пакет `v0`/`u0`** (имя меняется между версиями сборки, см.
  `bluetooth-protocol-trace-analysis.md` §7.1) — низкоуровневый Bluetooth SPP транспорт
  (класс `e`, UUID `00001101-0000-1000-8000-00805F9B34FB` — тот же, что использует наша
  библиотека `peripage`) и класс-контроллер команд протокола (класс `h`) — единственная
  точка, откуда уходят все сырые байтовые команды на принтер. Это основной объект разбора для
  протокольного документа.
- **Пакет `p9`** — параллельная/дублирующая реализация того же уровня (тоже classic
  Bluetooth SPP, тот же UUID, другой Log-тег `"PP"` вместо `"BluetoothPortPrint"`) — судя по
  всему, отдельная реализация для другого семейства устройств приложения (в `mvp/presenter/`
  видны отдельные модули для `businessLabel`, `epen`, `mbrush`, `tattoo`, `a4`, `plp`,
  `wrong` — приложение поддерживает намного больше устройств, чем термопринтеры-чеки).
  **Не разбирался подробно** — вне текущего скоупа (A40-класс принтеров), возможная точка
  расширения в будущем.
- **Пакет `g5`** — оркестрирующий слой печати: таблица ширин печати по модели/типу бумаги
  (обнаружено гораздо больше значений, чем 4 модели нашей библиотеки — 576/768/824/864/
  1236/1770/2120/2454.4/2496/2540 px), обработка изображений через OpenCV, и **реализация
  колбэков статуса принтера** (интерфейс `v0.c`, см. ниже).
- **Интерфейс `v0.c`** — асинхронные колбэки статуса принтера (батарея, крышка, перегрев,
  ошибка типа бумаги, прервать/продолжить печать), реализован в классе `g5.f` с осмысленными
  Log-тегами на китайском/английском (`"onOutPaper"`, `"onOverHeat"`, `"onLowVal"`,
  `"onStartOrStopSend"` и т.д.) — это раскрыло полную семантику статусных ответов, включая
  давнюю загадку входящих `fd01`/`fd02` из живого HCI-трейса (см. протокольный документ §4,
  §7.3).

**Важно для будущего повторного анализа**: между версией 312 и 319 обфускация переименовала
пакет `v0` в `u0` (класс с командами протокола) — побайтовое сравнение декомпилированного
кода (после нормализации переименованных полей) подтвердило, что **логика и все байтовые
константы идентичны**, изменились только сгенерированные имена. Искать по строковым
Log-тегам надёжнее, чем по именам классов/пакетов.
